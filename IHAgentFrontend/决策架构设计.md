# IHAgentFrontend 决策架构设计

> 状态：草案 v0.1
> 日期：2026-07-01
> 关系：本文档是 [CLAUDE.md](./CLAUDE.md) 在决策机制层面的细化补充。与 CLAUDE.md 冲突处以本设计为准，并标注需回写 CLAUDE.md 的修正点。

---

## 0. 设计原则

1. **LLM 做意图，规则做执行**：LLM 擅长自然语言理解和意图生成，不擅长实时空间/数值推理。规则承担确定性强、延迟敏感的决策。
2. **三层 LLM 配比差异化**：战略层 LLM 主导（意图生成）、调度层 LLM 主导（指令解析+任务拆解）、介入层规则主导（行为链执行）。介入层 LLM 按需介入。
3. **上下文即工作记忆**：决策结果作为"待执行事务"保留在 LLM 上下文里直到执行完毕，解决前后决策矛盾问题。
4. **玩家优先**：玩家指令任何时候可打断或覆盖 Agent 决策。

## 1. 整体架构与数据流

```
┌─────────────────────────────────────────────────────┐
│              玩家指令接口                              │
│  显式：自然语言（文字/语音）                          │
│  隐式：镜头位置 + Beacon放置                          │
└──────────────────┬──────────────────────────────────┘
                   │ 所有玩家指令（唯一入口）
                   ▼
┌──────────────────────────────────────────────────────┐
│              调度Agent [5s/事件]                       │
│  LLM主导 + 规则校验                                   │
│  ├ 解析玩家指令                                       │
│  ├ 识别战略级诉求 → 转发给战略Agent ──┐               │
│  └ 基于战略清单拆解为 Team 任务        │               │
│  输出：DispatchPlan                   │               │
└──────┬───────────────────────────────┴──────────────┘
       │ TeamTask                      ▲
       ▼                               │ 战略清单
┌────────────────────┐         ┌────────┴──────────────┐
│  介入Agent N        │         │  战略Agent [30s/事件] │
│  规则主导+LLM按需   │         │  LLM+规则态势预处理   │
│  行为链执行          │         │  输入：调度转发+态势  │
│  LLM三种调用模式     │         │  输出：StrategyDirective│
└────────────────────┘         └───────────────────────┘
       │
       ▼
   后端接口层（读态势/写指令/读反馈）
```

### 数据流概要

1. 玩家所有指令（显式自然语言 + 隐式镜头/Beacon）→ 调度Agent（唯一入口）
2. 调度Agent LLM 解析玩家指令：识别战略级诉求 → 转发给战略Agent；按紧急度选择时序路径（见 §2.3）
3. 战略Agent 接收调度转发的战略级诉求 + 规则预处理的态势摘要 → LLM 输出 StrategyDirective
4. StrategyDirective 回流到调度Agent → 调度Agent 基于战略清单 + 玩家指令拆解为 TeamTask
5. TeamTask 下发给对应 Team 的介入Agent → 介入Agent 执行行为链，按需唤醒 LLM
6. 介入Agent 执行反馈回流到调度Agent 和战略Agent

## 2. 玩家指令接口

### 2.1 指令输入通道

| 通道 | 形式 | 说明 |
|------|------|------|
| 显式 | 自然语言（文字/语音） | 战略指令、微操指令、模糊指令 |
| 隐式 | 镜头位置 | 玩家最近关注的区域，规则提取"关注区" |
| 隐式 | Beacon 放置 | 玩家在地图上放置的信号标记，规则提取"标记点" |

### 2.2 指令路由

**调度Agent 是唯一入口**。所有玩家指令先进调度Agent，调度Agent LLM 解析后识别战略级诉求，再转发给战略Agent。

> **修正点（需回写 CLAUDE.md）**：CLAUDE.md 第4节"战略指令→异步进入战略Agent"表述不准确，实际是"战略指令→调度Agent解析→转发给战略Agent"。

### 2.3 紧急度分流

玩家战略指令到达调度Agent 后，按紧急度选择时序路径：

| 紧急度 | 路径 | 说明 |
|--------|------|------|
| 紧急 | 先拆解后调整 | 调度Agent 立即做初步 TeamTask 拆解（不等战略清单），同时转发给战略Agent，战略清单到达后调整后续任务 |
| 非紧急 | 先转发等回流 | 调度Agent 转发给战略Agent，等 StrategyDirective 输出后再基于清单拆解 |

紧急度判定规则待定（候选：指令关键词如"立即/紧急"、指令来源如玩家连续点击同区域、战场态势如基地被攻击）。

## 3. 战略Agent

### 3.1 定位
军区司令。只输出"意图"，不输出"指令"。

### 3.2 唤醒条件
- 每 30 秒定时
- 调度Agent 转发的战略级诉求
- 调度Agent 上报重大事件

> **修正点（需回写 CLAUDE.md）**：CLAUDE.md 第1节战略Agent 唤醒条件含"玩家战略指令"，应改为"调度Agent转发的战略级诉求"。

### 3.3 输入

| 输入 | 来源 | 形式 |
|------|------|------|
| 历史操作链 | 自身历史 | 压缩摘要 |
| 战场态势 | 后端 + 规则预处理 | 结构化摘要（不喂生数据） |
| 上一轮输出 | 自身 | StrategyDirective |
| 调度反馈 | 调度Agent | 执行进度/损失/问题 |
| 战略级诉求 | 调度Agent 转发 | 玩家战略意图（结构化+自然语言片段） |

### 3.4 LLM 与规则分工

| 任务 | 执行者 |
|------|--------|
| 态势预处理（生数据→结构化摘要） | 规则 |
| 战略方向决策（直取/扩张/防守/骚扰） | LLM |
| 优先级权重 | LLM |
| 威胁评估 | 规则给数据，LLM 给判断 |
| 资源分配建议 | LLM |

### 3.5 输出：StrategyDirective

```cpp
struct StrategyDirective {
  Direction direction;          // 直取/扩张/防守/骚扰
  PriorityWeights priorities;   // 各战线权重
  ThreatAssessment threats;     // LLM 判断的威胁
  ResourceAllocation resources; // 资源分配建议
  GlobalObjective objective;     // 全局目标
  std::string rationale;        // 决策理由（便于调试和上下文延续）
};
```

## 4. 调度Agent

### 4.1 定位
前线指挥部。把战略拆解为具体任务，响应玩家指令和紧急事件。

### 4.2 唤醒条件
- 每 5 秒定时
- 新战略清单到达
- 玩家任何指令（唯一入口）
- Team 状态变化
- 紧急事件

### 4.3 输入
战略清单、态势（只读）、上一轮任务列表、介入Agent 状态、玩家指令（显式+隐式）、产线状态。

### 4.4 LLM 与规则分工

| 任务 | 执行者 |
|------|--------|
| 玩家指令解析（自然语言→意图） | LLM |
| 识别战略级诉求 | LLM |
| 战略指令时序路径选择 | 规则（基于紧急度判定） |
| 任务拆解（战略→TeamTask） | LLM |
| 行为链模板选择 | LLM（从模板库匹配） |
| 校验（防送死/越权/资源浪费） | 规则 |
| 产线安排 | LLM + 规则校验 |

### 4.5 输出：DispatchPlan

```cpp
struct DispatchPlan {
  std::vector<TeamTask> team_tasks;
  ProductionPlan production;
  EmergencyOverride emergency;   // 紧急覆盖（如基地被攻时全员回防）
  StrategyForward forward;      // 转发给战略Agent的战略级诉求
};

struct TeamTask {
  TeamId team;
  std::vector<UnitId> units;
  BehaviorTemplateId template_id;  // "offense_rush" / "defense_hold" / "llm_full_control"
  Target target;
  Priority priority;
  PlayerRequirement requirement;   // 玩家对该任务的具体要求（结构化+自然语言片段）
};
```

## 5. 介入Agent

### 5.1 定位
班长。带领指定单位执行具体任务，输出微操指令。规则主导，LLM 按需介入。

### 5.2 行为链模板

自研的、预定义的、无变通性。每条行为链由若干节点和分支点组成。

```cpp
struct BehaviorChain {
  std::string template_id;              // "offense_rush"
  std::vector<BehaviorNode> nodes;       // 顺序节点
  std::vector<BranchPoint> branches;    // 分支点
};

struct BehaviorNode {
  Action action;        // gather_at_base / move_to / request_support / attack / ...
  Params params;
};

struct BranchPoint {
  int at_node;                      // 在第几个节点后触发
  Trigger trigger;                  // enemy_contact / support_unavailable / losses_exceed / ...
  RuleDecision rule;                // 规则尝试决策
  Fallback fallback;                // rule / llm —— 规则无法决断时回退到谁
};
```

示例（进攻行为链）：

```
offense_rush:
  nodes:
    0: gather_at_base(radius=5)
    1: move_to(target=enemy_base, formation=column)
    2: request_support(type=iron_curtain)
    3: attack(target=enemy_base, mode=aggressive)
  branches:
    {at_node:1, trigger:enemy_contact, rule:decide_engage_or_bypass, fallback:llm}
    {at_node:2, trigger:support_unavailable, rule:decide_proceed_or_wait, fallback:rule}
```

### 5.3 分支点决策机制

行为链执行到分支点时：
1. 触发条件满足 → 进入分支判定
2. 规则尝试决策（rule 字段）
3. 规则能决断 → 规则给出决策，行为链继续
4. 规则无法决断 且 `fallback=llm` → 挂起行为链，唤醒 LLM（默认进入"分支决策"模式）

### 5.4 LLM 三种调用模式

| 模式 | 触发 | 状态 | 输入 | 输出 | 调用频率 |
|------|------|------|------|------|----------|
| **分支决策** | 行为链分支点规则难决断 | 无状态 | 当前状态+可选动作+玩家要求 | 单个决策 | 单次 |
| **接管模式** | 无模板 / 连续3次分支冲突 | 有状态（Team级上下文） | 同上+历史决策+时间窗口 | 持续决策序列 | 每 Δt 一次，持续至危机解除或上限 M 秒 |
| **行为校验** | 玩家有具体要求时 | 无状态 | 拟执行动作+玩家要求 | allow/deny/modify | 单次 |

**升级机制**：同一 Team 在时间窗口 T 内连续触发 N=3 次"分支决策"模式 → 升级为接管模式。接管模式持续至危机解除（危机判定规则待定）或时间上限 M 秒后归还规则。

**降级机制**：接管模式中，若连续 K 次决策均为常规分支（无新危机）→ 降级回单次分支决策模式。K 值待定。

### 5.5 无模板场景

调度Agent 下发的 `TeamTask.template_id = "llm_full_control"` 时，介入Agent 直接进入接管模式，由 LLM 全权负责该 Team 的微操。此模式下 LLM 临时兼任"调度+介入"角色，但权限仍受 TeamTask 约束（只能操作分配的单位，受 PlayerRequirement 约束）。

### 5.6 输出

```cpp
struct InterventionOutput {
  std::vector<MicroOrder> orders;   // Move/Attack/Guard/...
  ExecutionFeedback feedback;      // 进度/损失/问题
};
```

## 6. 上下文管理

### 6.1 上下文即工作记忆
- 每个 LLM 调用（战略/调度/介入层）维护独立上下文
- 决策结果作为"待执行事务"保留在上下文里，直到事务执行完毕（成功/失败/取消）才清理
- 后续决策可引用上下文中的未完成事务，避免前后矛盾

### 6.2 跨层上下文
- 战略Agent ↔ 调度Agent：通过 StrategyDirective 和 DispatchFeedback 显式传递，不共享上下文
- 调度Agent ↔ 介入Agent：通过 TeamTask 和 ExecutionFeedback 显式传递，不共享上下文
- 介入Agent 接管模式内部：维护 Team 级上下文（历史决策+态势变化）

> **待确认**：战略Agent 和调度Agent 是否需要共享部分上下文以减少重复传递？目前设计为不共享，靠显式数据传递。

## 7. 关键设计决策记录（ADR）

| ID | 决策 | 选择 | 理由 | 状态 |
|----|------|------|------|------|
| ADR-01 | 三层 LLM 配比 | 战略LLM主导 / 调度LLM主导 / 介入规则主导 | 各层用最适合的方式，LLM 擅长意图不擅长实时推理 | 已确认 |
| ADR-02 | 介入规则实现 | 自研行为链模板 | 可控、可调、可解释 | 已确认 |
| ADR-03 | LLM 介入粒度 | 默认单次 + 连续3次冲突升级短时接管 | 平衡连贯性与可控性 | 已确认 |
| ADR-04 | 玩家指令输入 | 显式自然语言 + 隐式镜头/Beacon | 隐式补战斗中无法打字场景 | 已确认 |
| ADR-05 | 玩家指令路由 | 调度Agent 唯一入口 | 统一入口便于指令解析和路由 | 已确认（需回写CLAUDE.md） |
| ADR-06 | 战略指令时序 | 按紧急度分流 | 紧急要响应快，非紧急要准确 | 已确认 |
| ADR-07 | 态势预处理 | 规则做摘要 | 避免 LLM 啃生数据，省 token | 已确认 |
| ADR-08 | 上下文管理 | 上下文即工作记忆 | 解决前后矛盾 | 已确认 |
