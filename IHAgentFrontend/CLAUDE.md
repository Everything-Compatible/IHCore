# IHAgentFrontend — 项目说明

> 决策机制的详细设计见 [决策架构设计.md](./决策架构设计.md)。

## 项目定位

IHAgentFrontend 是 IHCore 引擎扩展的 **AI Agent 前端层**，负责游戏内的智能决策与操控逻辑。

根据 [Agent架构.md](../Agent架构.md) 的划分，整个系统分为：

- **后端**（本项目不涉及）：共享状态 (GameState)、执行层（原版引擎/脚本）
- **前端**（本项目负责）：战略Agent、调度Agent、介入Agent、玩家指令接口

---

## 范围（Scope）

### 本项目负责

| 模块 | 定位 | 说明 |
|------|------|------|
| **战略Agent** (Strategic Agent) | 军区司令 | LLM 主导（意图生成），方向性决策，每30秒评估战场态势，输出战略清单 |
| **调度Agent** (Dispatch Agent) | 前线指挥部 | LLM 主导（指令解析+任务拆解）+ 规则校验，玩家指令唯一入口 |
| **介入Agent** (Intervention Agent) | 班长 | 规则主导 + LLM 按需介入，行为链执行，输出微操指令 |
| **玩家指令接口** (Player Command Interface) | 输入层 | 显式（自然语言）+ 隐式（镜头位置/Beacon放置），统一进入调度Agent |

### 本项目不负责

- 共享状态 (GameState) 的存储与维护 —— 由后端提供只读/读写接口
- 微操指令的实际执行（IssueOrder/QueueMission）—— 由后端的执行层处理
- 原版引擎的索敌、避障、威胁评估 —— 内置在引擎中，零Token消耗

## 模块职责

### 1. 战略Agent (Strategic Agent)

- **定位**：军区司令。LLM 主导（意图生成），规则负责态势预处理。
- **唤醒条件**：每30秒定时 / 调度Agent转发的战略级诉求 / 调度Agent上报重大事件
- **输入**：历史操作链（压缩摘要）、战场态势（规则预处理的结构化摘要）、上一轮输出、调度反馈、战略级诉求（调度Agent转发）
- **输出**：`StrategyDirective` — 包含方向(直取/扩张/防守/骚扰)、优先级权重、全局目标、威胁评估、资源分配建议、决策理由
- **原则**：只输出"意图"，不输出"指令"

**LLM 与规则分工**：

| 任务 | 执行者 |
|------|--------|
| 态势预处理（生数据→结构化摘要） | 规则 |
| 战略方向决策（直取/扩张/防守/骚扰） | LLM |
| 优先级权重 | LLM |
| 威胁评估 | 规则给数据，LLM 给判断 |
| 资源分配建议 | LLM |

### 2. 调度Agent (Dispatch Agent)

- **定位**：前线指挥部。LLM 主导（指令解析+任务拆解），规则校验。
- **唤醒条件**：每5秒定时 / 新战略清单到达 / 玩家任何指令（唯一入口）/ Team状态变化 / 紧急事件
- **输入**：战略清单、战场态势（只读）、上一轮任务列表、介入Agent状态、玩家指令（显式+隐式）、产线状态
- **输出**：`DispatchPlan` — 包含Team任务列表（行为链模板ID、目标、优先级、玩家要求）、产线安排、紧急覆盖、转发给战略Agent的战略级诉求

**LLM 与规则分工**：

| 任务 | 执行者 |
|------|--------|
| 玩家指令解析（自然语言→意图） | LLM |
| 识别战略级诉求 | LLM |
| 战略指令时序路径选择 | 规则（基于紧急度判定） |
| 任务拆解（战略→TeamTask） | LLM |
| 行为链模板选择 | LLM（从模板库匹配） |
| 校验（防送死/越权/资源浪费） | 规则 |
| 产线安排 | LLM + 规则校验 |

### 3. 介入Agent (Intervention Agent)

- **定位**：班长。规则主导（行为链执行），LLM 按需介入。
- **唤醒条件**：调度Agent下发TeamTask / 玩家指令经调度转发 / 行为链分支点触发LLM回退
- **输入**：TeamTask（含行为链模板ID、分配单位、目标、优先级、玩家要求）、本队周边态势
- **输出**：微操指令序列（Move/Attack/Guard等）+ 执行反馈（进度/损失/问题）
- **原则**：仅操作分配的单位，受TeamTask约束

**行为链模板**：自研的、预定义的执行模板。每条行为链由顺序节点和分支点组成。分支点触发时规则优先尝试决策；规则无法决断时，按 fallback 设置回退到 LLM 或规则默认处理。示例模板：`offense_rush`、`defense_hold`、`llm_full_control`。

**LLM 三种调用模式**：

| 模式 | 触发条件 | 状态 | 说明 |
|------|----------|------|------|
| 分支决策 | 行为链分支点规则难决断 | 无状态 | 单次调用，输入当前状态+可选动作+玩家要求，输出单个决策 |
| 接管模式 | 无模板 / 连续N次分支冲突 | 有状态（Team级上下文） | 持续决策序列，每Δt一次，直至危机解除或M秒上限后归还规则 |
| 行为校验 | 玩家有具体要求时 | 无状态 | 单次调用，输入拟执行动作+玩家要求，输出 allow/deny/modify |

> 详细设计见 [决策架构设计.md §5](./决策架构设计.md#5-介入agent)。

### 4. 玩家指令接口

- **路由规则**：调度Agent 是玩家指令的唯一入口。所有指令（显式+隐式）先进调度Agent，调度Agent LLM解析后识别战略级诉求，再转发给战略Agent。
- **指令输入通道**：

| 通道 | 形式 | 说明 |
|------|------|------|
| 显式 | 自然语言（文字/语音） | 战略指令、微操指令、模糊指令 |
| 隐式 | 镜头位置 | 玩家最近关注的区域，规则提取"关注区" |
| 隐式 | Beacon 放置 | 玩家在地图上放置的信号标记，规则提取"标记点" |

- **紧急度分流**：紧急指令走"先拆解后调整"路径（不等战略清单），非紧急指令走"先转发等回流"路径。
- **核心原则**：玩家指令任何时候都要立即响应，不能等定时周期。

## 外部接口（对接后端）

本项目通过以下接口与后端（共享状态 + 执行层）交互：

| 接口方向 | 用途 | 频率 |
|---------|------|------|
| **读共享状态** | 查询战场态势、单位状态、产线状态 | 各Agent按自身周期 |
| **写微操指令** | 通过后端执行层下发 IssueOrder/QueueMission | 事件驱动 |
| **读执行反馈** | 获取指令执行结果 | 事件驱动 |

具体接口定义由后端提供，本项目遵循其约定。

## 关键原则

1. **LLM 做意图，规则做执行**：LLM 擅长自然语言理解和意图生成，不擅长实时空间/数值推理。规则承担确定性强、延迟敏感的决策。
2. **三层 LLM 配比差异化**：战略层 LLM 主导（意图生成）、调度层 LLM 主导（指令解析+任务拆解）、介入层规则主导（行为链执行），介入层 LLM 按需介入。
3. **上下文即工作记忆**：决策结果作为"待执行事务"保留在 LLM 上下文里直到执行完毕，解决前后决策矛盾问题。
4. **玩家优先**：玩家指令任何时候可打断或覆盖 Agent 决策。
5. **职责分离**：战略Agent只出意图，调度Agent拆解任务，介入Agent执行微操。
6. **权限约束**：介入Agent只能操作分配的单位，不得越权。
7. **零副作用的读操作**：战略Agent和态势查询均为只读。

---

## 上下文管理

- 每个 LLM 调用（战略/调度/介入层）维护独立上下文，跨层通过结构化数据显式传递，不共享上下文。
- 决策结果作为"待执行事务"保留在上下文里，直到事务执行完毕（成功/失败/取消）才清理。后续决策可引用未完成事务，避免前后矛盾。
- 介入Agent 接管模式内部维护 Team 级上下文（历史决策+态势变化）。

> 详细设计见 [决策架构设计.md §6](./决策架构设计.md#6-上下文管理)。

## 开发约定

- 使用 C++17/20 标准
- 每个 Agent 对应独立的源文件或模块
- Agent 间的通信优先通过接口/事件回调，避免直接耦合
- 所有 Agent 的输出数据结构化（JSON 或对应的 C++ 结构体）
